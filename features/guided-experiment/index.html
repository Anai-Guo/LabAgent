    Guided Experiment (Web)

    The /experiment page is the fully-automated Web flow: enter your research direction and sample material, then let LabAgent handle literature search, instrument scanning, AI-based measurement decision, simulated data collection, and literature-aware analysis. It is the recommended entry point for Web users and replaces the interactive CLI prompts with a clickable, streaming interface.

    Opening the Page

    Start the web server and open the guided experiment URL:

    labharness web --port 8080

    Then navigate to http://localhost:8080/experiment in your browser. The Dashboard (/) and Monitor (/monitor) are still available for manual template configuration — use /experiment when you want the AI to pick the measurement type for you.

    UI Layout

    The page opens with a centered start card asking two questions: Research Direction (e.g. transport, photovoltaics, magnetism) and Sample Material (e.g. Si wafer, CoFeB thin film, GaAs). Click Start Experiment and the layout switches to a two-column view.

    • Left column — Live Activity log. A dark, monospace stream of everything the backend is doing: literature search progress, VISA resource IDs found, role classifications, phase transitions. Lines are color-coded (gold for phase headers, cyan for instruments, violet for literature hits, green for roles). This is your audit trail.
    • Right column — Sequential result cards. Cards appear from top to bottom as each phase of the flow completes. You interact with the cards to continue the flow; the activity log keeps streaming in parallel.

    The page uses Server-Sent Events (/api/experiment/{session_id}/events) so everything updates without page reloads.

    The Five Cards

    Cards appear in this order, each one triggered by a phase event from the orchestrator:

    1. AI Decision card. Shows the chosen measurement_type in gold (e.g. AHE, IV, MR) and a reasoning block from the LLM explaining why this type was picked based on your direction, your connected instruments, and the literature context. If no AI model is configured, a rule-based fallback is used and the reasoning block explains which heuristic fired.
    2. Save Data To card. Pre-fills a suggested folder name derived from the session ID, date, measurement type, and material. You can edit it, and optionally change the Parent Directory (defaults to ./data). The server sanitizes slashes and path traversal, and only accepts parents under the current working directory or your home directory. Click Continue to confirm.
    3. Measurement Progress card. Shows a gold progress bar, a point N / total text counter, and a small sparkline canvas that updates as each point streams in. Because this is the simulated mode, points arrive quickly; a physics-based simulator (see orchestrator/simulators.py) generates realistic data shaped by your material and literature context.
    4. Results card. Displays generated figures (PNG previews fetched from /api/experiment/{session_id}/figure/{name}), an extracted-values table with key physical quantities from the analyzer, and an AI Interpretation block. The interpretation is a paragraph-length explanation produced by the LLM that cites retrieved papers inline using [N] markers (see below).
    5. Suggested Next Steps card. A short markdown list of 3 concrete follow-up experiments. This section only appears if the LLM is configured. The text is also written to next_steps.md inside your data folder.

    A final Open Data Folder card appears at the end with a single button that opens the output directory in your OS file explorer.

    What "SIMULATED" Means

    Between the Measurement and Results cards you will see a SIMULATED banner in the activity log, and the progress card note says data is generated by the physics simulator. This is intentional in the current release:

    • No commands are sent to your connected instruments — the scan runs only to inform the AI decision, not to drive hardware.
    • Data is generated by orchestrator.simulators.simulate(plan, measurement_type, seed=42, literature=...), which produces deterministic, physically-plausible curves (e.g. an AHE loop with a coercive field, an IV curve with the expected non-linearity) influenced by the retrieved literature.
    • Output files, folder layout, analysis pipeline, and figure generation are real and identical to what you will see when driver integration lands — so your downstream scripts and notebooks keep working.

    Real-driver execution is on the roadmap (PyMeasure integration). When enabled it will replace the simulator step without changing the rest of the flow.

    Reading [N] Citations

    The AI Interpretation block embeds citation markers like [1], [2] that refer to papers retrieved during the parallel literature search. The matching bibliography appears in the Literature Referenced card: each entry is a clickable title with authors, year, and source (DOI or arXiv ID). Hovering or clicking [N] in the interpretation scrolls the card into view so you can cross-check the claim against the source paper. The same [N] indexing is preserved in the summary.json written to disk, so citations survive outside the browser.

    Where Data is Saved

    Each run writes to <parent_dir>/<folder_name>/ with this layout:

    • raw_data.csv — the measurement table, one row per sweep point, columns from your plan's x_axis and y_channels.
    • <type>_analysis.py — the exact analysis script that ran (template or AI-generated), reproducible with python <script>.
    • fig_*.png and fig_*.pdf — publication-ready figures at 300 dpi.
    • summary.json — the frozen session state: direction, material, plan, validation, extracted values, interpretation with [N] markers, literature bibliography, timings.
    • next_steps.md — optional follow-up suggestions from the LLM.

    The folder name is pre-suggested as YYYYMMDD_HHMMSS_<type>_<material> but you can override it at the folder-confirmation step. Click Open Data Folder at the end to jump straight to it, or browse manually — everything is a flat directory of plain files, no database.

    Troubleshooting

    • Page opens but the start button does nothing. Check the browser console for CORS or 404 errors on /api/experiment/start_async. Reloading the page creates a fresh session; finished sessions are auto-cleaned up one hour after completion.
    • AI Decision card never appears. The literature-search step is timing out. The activity log will show a warning. The flow continues with an empty literature context and the rule-based fallback picks the measurement type.
    • Measurement Progress bar stalls. Check the server log for simulator exceptions. The simulator is deterministic, so the same direction + material always reproduces the same stall — paste the log into an issue.
    • Figures card shows broken images. The analyzer script failed after generating the CSV. Open the data folder and run the <type>_analysis.py script manually to see the traceback; the raw data is still valid.
    • Server memory growth over long sessions. The session registry auto-evicts sessions that completed more than one hour ago. To tune, construct SessionRegistry(done_ttl_seconds=...) with a different TTL.
